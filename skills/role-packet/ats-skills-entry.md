# Application skills entry list

Most online applications (Workday, Greenhouse, Taleo, …) make you enter a **discrete list of skills**
separate from the resume — a form field, a tag picker, or a "key skills" box the ATS parses and
recruiters filter on. This artifact produces that list so it's deliberate and keyword-aligned, not
improvised at 11pm in the portal.

## How to build it

1. Pull the **exact skill terms** from the posting (required first, then preferred). Use the posting's
   own wording where an ATS is likely doing literal keyword matching (e.g. "Product Lifecycle
   Management", not your synonym for it).
2. Keep only skills the **evidence map** backs as Direct or defensibly Adjacent. If the candidate
   can't substantiate it in an interview, it does not go on the list — a padded skills list is a
   liability the moment a screener asks about it.
3. **Rank** by (a) how central the skill is to the posting and (b) strength of evidence. Lead with the
   ones that are both required and provable.
4. Note proficiency honestly where the form asks for it (expert / proficient / familiar), matched to
   the source of truth's skills inventory.

## Format (write to `application skills entry list.md`)

```
# Application skills entry list — <Role> @ <Company>

## Enter these (ranked)
1. <exact skill term>  — proficiency: <expert/proficient/familiar> — evidence: <role/story>
2. …

## Preferred / nice-to-have (if the form has room)
- <term> — evidence: …

## Deliberately omitted (would not survive a screen)
- <term the JD lists but the candidate can't substantiate> — reason: …
```

## Guardrails
- Match the posting's phrasing for literal-match ATS fields; don't "improve" a term the parser is
  looking for verbatim.
- The **omitted** section is not filler — it's the record of what you chose not to claim, so the
  decision is visible to the user and consistent with the resume and cover letter.
